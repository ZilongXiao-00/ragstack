# 架构设计文档

## 系统架构

RAGStack 采用模块化设计，分为四个核心层次：

```
┌─────────────────────────────────────┐
│           应用层 (Application)        │
│         RAGPipeline 主流程           │
└─────────────┬───────────────────────┘
              │
┌─────────────▼───────────────────────┐
│           编排层 (Orchestration)      │
│    RetrievalOrchestrator 检索协调器   │
└─────────────┬───────────────────────┘
              │
┌─────────────▼───────────────────────┐
│           能力层 (Capabilities)       │
│  ┌─────────┬─────────┬───────────┐  │
│  │ 索引构建 │ 检索策略 │  LLM生成  │  │
│  │Indexing │Retrieval│Generation │  │
│  └─────────┴─────────┴───────────┘  │
└─────────────┬───────────────────────┘
              │
┌─────────────▼───────────────────────┐
│           基础设施层 (Infrastructure)  │
│  ChromaDB │ BGE模型 │ BM25 │ jieba  │
└─────────────────────────────────────┘
```

## 核心流程

### 1. 索引构建流程

```
文档输入
    │
    ▼
父子分片 (Chunking)
    │
    ├─ 父块 (1000字符) → 用于Embedding检索
    └─ 子块 (300字符) → 存储在父块元数据中
    │
    ▼
向量编码 (BGE Embedding)
    │
    ▼
存入 ChromaDB
```

### 2. 检索流程

```
用户查询
    │
    ├─→ 语义检索 ──┐
    │              ├──→ RRF融合 ──→ 父块重排 ──→ Top-M父块
    └─→ BM25检索 ──┘
                            │
                            ▼
                    提取子块 → 子块重排 → Top-N子块
                            │
                            ▼
                        LLM生成回答
```

## 关键技术决策

### 为什么选择父子分片？

| 粒度 | 用途 | 优势 |
|-----|------|------|
| 父块(大) | 检索 | 语义完整，召回率高 |
| 子块(小) | 生成 | 内容精准，噪声少 |

### 为什么使用RRF融合？

- 语义检索：擅长理解意图，但可能漏掉专有名词
- BM25检索：擅长精确匹配，但缺乏语义理解
- RRF融合：结合两者优势，不依赖分数量纲

### 为什么双层重排？

1. **父块重排**：从召回的多个父块中筛选最相关的
2. **子块重排**：从选中父块的子块中筛选最相关的
3. **优势**：逐步缩小范围，提高最终精度
